## Context

当前仓库在 `src/strategies/` 下只定义了一个最小 `BaseStrategy` 接口，还没有任何正式策略实现、参数约束或测试样例。`docs/ai-quant/strategies/AI量化-趋势策略-MACD.md` 已经描述了 MACD 的业务语义，但这些研究文档尚未转化为代码契约。

这次设计需要在不引入完整交易系统复杂度的前提下，先落地一个可测试、可回测接入的趋势策略实现。仓库当前缺少成熟的因子层和回测层，因此设计必须尽量收敛到最小闭环，避免把 MACD 策略与未来尚未存在的执行框架强耦合。

## Goals / Non-Goals

**Goals:**
- 定义一个与 `BaseStrategy` 兼容的 MACD 趋势策略实现，能够从标准行情表中生成信号与目标仓位。
- 明确 MACD 所需输入列、默认参数、输出字段和缺失数据处理规则，避免后续实现出现隐式假设。
- 将基础风控约束纳入策略接口，例如空仓、单方向持仓和趋势过滤，保证策略行为可预测。
- 为后续回测或脚本接入预留稳定的调用方式，而不是把逻辑写死在 notebook 或一次性脚本中。

**Non-Goals:**
- 不实现完整的订单执行、经纪商对接或实盘调度流程。
- 不在本次设计中引入多标的组合优化、资金管理模型或复杂仓位缩放逻辑。
- 不要求实现独立的通用技术指标库；若需要，可先在 MACD 策略内部封装最小计算逻辑。
- 不承诺本次变更同时交付完整回测引擎，只要求策略能被后续回测链路调用。

## Decisions

### Decision: 使用单一 DataFrame 输入和显式列契约
MACD 策略将接受按时间升序排列的表格数据，至少包含 `close` 列；可选包含时间索引或 `datetime` 列。策略内部只依赖收盘价计算 EMA、DIF、DEA 和柱状图，并在输出中保留原始索引。

Rationale:
- 现有 `BaseStrategy` 接口没有定义更复杂的数据对象，延续 DataFrame 风格接入成本最低。
- MACD 的核心计算只依赖收盘价，避免在第一版引入不必要的 OHLCV 约束。
- 显式列契约便于测试和后续脚本复用。

Alternatives considered:
- 自定义 `MarketData` 类：更规范，但对当前代码基线来说过重。
- 直接接收 `Series`：过于轻量，不利于后续扩展附加列和信号输出。

### Decision: 将默认策略规则限定为金叉开多、死叉平多，并支持可选零轴过滤
第一版策略默认只做多，不做空。买入信号基于 DIF 上穿 DEA，卖出信号基于 DIF 下穿 DEA。策略参数中允许启用零轴过滤，例如仅在 DIF 大于等于 0 时接受开多信号。

Rationale:
- 仓库文档里对 MACD 的核心认知以趋势跟随为主，先做单向多头策略最符合当前项目阶段。
- 只做多可以避免在没有做空约束、借券和保证金语义之前引入错误抽象。
- 零轴过滤是常见增强规则，适合作为参数而非硬编码行为。

Alternatives considered:
- 同时支持做多做空：能力更强，但会把仓位状态和风控复杂度翻倍。
- 仅输出买卖布尔信号，不输出仓位：不利于后续回测直接消费。

### Decision: 输出标准化的信号与仓位结果表
`generate_signals` 应输出包含至少 `dif`、`dea`、`histogram`、`signal` 的结果表，其中 `signal` 使用明确枚举值表示 `buy`、`sell` 或 `hold`。`build_positions` 基于该信号表输出目标仓位，默认只允许 `0` 和 `1` 两种状态。

Rationale:
- 技术指标值本身对调试、测试和研究都很关键，不能只保留最终交易动作。
- 明确的枚举值和离散仓位更便于编写断言。
- 0/1 仓位模型足以覆盖第一版 MACD 趋势策略。

Alternatives considered:
- 只输出最终仓位：会丢失中间指标，不利于排障。
- 允许连续仓位：当前没有明确的仓位管理规则支撑。

### Decision: 缺失数据和预热窗口期间强制空仓
在 EMA 尚未稳定、输入为空、缺少 `close` 列、或出现无法计算的缺失段时，策略必须避免产生买入仓位。预热期和无效期内 `signal` 输出为 `hold`，目标仓位为 `0`。

Rationale:
- 趋势策略最常见的回测错误之一是把预热期或缺失值误判为有效信号。
- 对第一版实现而言，保守地回退到空仓比隐式补值更安全。

Alternatives considered:
- 自动前向填充所有缺失价格：会掩盖数据问题并扭曲指标。
- 在异常时直接抛错终止：对研究和批量回测不够友好。

## Risks / Trade-offs

- [Risk] 第一版只做多，无法覆盖下跌趋势中的反向收益场景。 -> Mitigation: 在 spec 中明确单向持仓范围，后续如需做空再新增 delta。
- [Risk] 将指标计算放在策略模块内部，短期内会与未来通用因子层存在重复。 -> Mitigation: 保持计算函数独立、纯函数化，后续可平滑迁移到 `src/factors/`。
- [Risk] 当前仓库没有成熟回测引擎，策略输出格式若定义不清会导致后续集成返工。 -> Mitigation: 在 spec 中明确输入列、信号枚举和目标仓位语义。
- [Risk] MACD 在震荡市容易频繁反转，若没有额外过滤条件，回测结果可能不稳定。 -> Mitigation: 将零轴过滤和参数化能力纳入规格，保留后续优化空间。

## Migration Plan

1. 在 `src/strategies/trend/` 新增 MACD 策略模块，并对外暴露可实例化类。
2. 如有必要，微调 `src/strategies/base.py` 的文档或最小接口说明，但保持向后兼容。
3. 新增测试夹具和测试用例，覆盖指标计算、信号切换和异常输入。
4. 为研究或回测补充最小调用入口，验证策略可被外部模块消费。
5. 若实现不符合预期，可通过移除新增策略模块和测试文件回滚，不影响现有空骨架结构。
