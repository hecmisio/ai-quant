## Context

根据 [AI量化-Anne数据库选型决策](/D:/ai-quant/docs/ai-quant/AI量化-Anne数据库选型决策.md)，Anne 的主库路线已经确定为 `PostgreSQL + TimescaleDB + pgvector`。当前仓库缺少可直接复用的数据库基础设施定义，导致后续表结构、迁移、数据入库和本地联调都没有统一起点。

这次设计聚焦在“数据库启动基线”而不是业务数据模型本身。目标是在仓库内提供一套可以稳定启动的 PostgreSQL 18.3 开发环境，并在首次初始化时启用 Anne 后续会依赖的扩展。

约束包括：

- 版本基线固定为 `PostgreSQL 18.3`，以拉长新项目生命周期。
- 设计必须兼容 `TimescaleDB` 与 `pgvector` 的启用方式。
- 实现应尽量保持本地开发简单，不引入额外编排系统。
- 初始化过程应可重复理解，但首次建库时自动完成扩展准备。

## Goals / Non-Goals

**Goals:**

- 为仓库提供统一的本地数据库启动入口。
- 固定 Anne 主库的 PostgreSQL 18.3 基线和基础运行参数。
- 在数据库初始化阶段自动启用 `timescaledb` 与 `vector` 扩展。
- 为后续 schema、迁移脚本和数据接入保留清晰目录约定。

**Non-Goals:**

- 不在本次设计中定义业务表结构、索引或 hypertable 细节。
- 不实现生产环境部署、备份恢复或高可用架构。
- 不引入 ClickHouse、MinIO 或其他后续分层组件。
- 不替代未来正式迁移工具的职责；本次仅建立开发启动基线。

## Decisions

### 1. 使用 Docker Compose 作为本地数据库启动入口

选择在仓库根目录提供 `docker-compose.yml`，而不是要求开发者手动安装数据库。

原因：

- 开发环境启动方式统一，降低团队成员接入成本。
- Compose 足以覆盖当前单节点开发场景，复杂度明显低于 Kubernetes 或额外脚本封装。
- 后续如需增加 Admin 工具、备份容器或测试依赖，也容易在同一编排文件中扩展。

备选方案：

- 本机直接安装 PostgreSQL 与扩展：不利于版本锁定和团队一致性。
- 自定义启动脚本：仍然要处理镜像、卷、端口和初始化挂载，实际维护成本更高。

### 2. 锁定 PostgreSQL 18.3 作为镜像基线

选择显式锁定到 `18.3`，而不是使用 `18` 或 `latest` 这类浮动标签。

原因：

- 小版本固定可以降低“同一仓库、不同时间启动得到不同数据库行为”的风险。
- 该项目明确希望拉长生命周期，因此应从最新稳定主版本开始，但避免无提示漂移。

备选方案：

- `postgres:18`：会随上游镜像更新，削弱可重复性。
- `postgres:17.x`：更保守，但不符合当前“尽量拉长生命周期”的优先级。

### 3. 通过初始化脚本启用 TimescaleDB 与 pgvector

选择将扩展启用逻辑放入容器初始化目录，在首次建库时自动执行。

原因：

- 扩展安装和启用是 Anne 主库后续数据模型的前置条件。
- 将初始化 SQL 放在仓库中，可审阅、可版本管理，也便于后续追加初始化逻辑。
- 与手工执行 `CREATE EXTENSION` 相比，一次性进入开发基线更稳妥。

备选方案：

- 手工连接数据库后执行扩展语句：容易遗漏，无法保证环境一致。
- 后续再由迁移工具创建扩展：可以作为长期方案，但当前连基础库都尚未可启动，需要先建立最低可用基线。

### 4. 本次只定义基础环境变量、卷和初始化目录，不预建业务 schema

选择仅提供数据库服务本身及扩展初始化，不在此阶段加入具体业务表。

原因：

- 当前变更的目标是为后续实现提供基线，而不是提前冻结业务模型。
- 表结构将直接受到后续 OpenSpec 变更和 Anne 核心数据域划分影响，现在提前落库会增加返工概率。

备选方案：

- 同时预建 `market_data`、`news`、`events` 等 schema：会把基础设施变更与数据建模耦合在一起，职责不清。

## Risks / Trade-offs

- [TimescaleDB for PostgreSQL 18 镜像/包选择不当] -> 优先选用已明确支持 PG18 的发行方式，并在实现时锁定具体 tag。
- [初始化脚本只在首次建库执行，后续修改不会自动重跑] -> 将脚本定位为“基线初始化”，后续结构变更交给正式迁移机制。
- [本地端口或卷命名与开发者现有环境冲突] -> 在 Compose 中暴露可配置环境变量，并采用项目作用域内的服务命名。
- [开发基线与未来生产部署存在差异] -> 在设计中明确本次范围仅限本地/开发基线，不把 Compose 直接等同于生产方案。
