## 背景

当前仓库已经具备 `docs`、`src`、`openspec` 和 `third` 等基础目录，但项目尚未将“研究文档、数据资产、策略代码、回测执行、配置与输出”之间的边界固化为正式规范。随着后续开始开发具体策略，如果目录结构缺乏统一标准，容易出现研究笔记、实验脚本、正式实现和运行产物混放的情况，增加维护成本，也削弱可复现性。

本次变更用于把完整的项目目录规划上升为项目级 spec，作为后续策略开发、回测实现和实盘接入的共同基线。

## 变更内容

- 在 `project-governance` capability 中补充项目目录结构标准。
- 明确研究文档、可执行代码、数据、配置、测试、脚本和运行输出的目录职责边界。
- 明确 `src` 作为正式代码区，`docs` 作为研究与说明区，避免职责混淆。
- 为后续策略、回测、风控和执行能力提供一致的落位约定。
- 明确 `data` 与 `outputs` 默认不提交 Git，仅允许为测试或复现目的保留极小规模样本。

推荐的完整项目目录结构如下：

```text
ai-quant/
├─ docs/                      # 你的策略思考、市场认知、专题笔记
├─ data/
│  ├─ raw/                    # 原始行情、财务、新闻等，不做修改
│  ├─ interim/                # 清洗中间结果
│  ├─ processed/              # 回测/训练直接使用的数据
│  └─ snapshots/              # 关键时点快照，便于复现
├─ notebooks/                 # 临时研究、指标验证、想法试验
├─ config/
│  ├─ env/                    # dev/test/prod 配置
│  ├─ data/                   # 数据源配置
│  └─ strategies/             # 各策略参数配置
├─ src/
│  ├─ common/                 # 通用工具：时间、日志、文件、配置加载
│  ├─ data/                   # 数据接入、清洗、特征加工
│  ├─ factors/                # 因子计算、技术指标、信号组件
│  ├─ strategies/             # 策略实现
│  │  ├─ trend/               # 趋势类
│  │  ├─ mean_reversion/      # 均值回归/震荡类
│  │  ├─ rotation/            # 轮动类
│  │  └─ base.py              # 策略基类/统一接口
│  ├─ portfolio/              # 仓位管理、组合构建、资金分配
│  ├─ risk/                   # 止损、限仓、风控规则
│  ├─ backtest/               # 回测引擎、撮合、绩效分析
│  ├─ execution/              # 实盘下单、券商/交易接口适配
│  └─ pipelines/              # 把“取数→算信号→回测/执行”串起来
├─ tests/
│  ├─ unit/
│  ├─ integration/
│  └─ fixtures/
├─ scripts/                   # 命令行脚本，如批量回测、数据更新
├─ outputs/
│  ├─ reports/                # 回测报告
│  ├─ signals/                # 每日信号输出
│  └─ logs/                   # 运行日志
├─ third/
│  ├─ docs/                   # 第三方官方文档、接口手册、供应商资料
│  ├─ sdk/                    # 第三方 SDK、安装包或无法通过包管理器获取的依赖
│  └─ examples/               # 第三方官方示例
└─ README.md
```

其中：
- `docs` 用于沉淀研究与认知，不承担正式实现职责。
- `src` 用于正式代码实现，并在内部继续按量化系统职责拆分模块。
- `data` 与 `outputs` 默认不提交 Git，仅保留目录结构或极小规模复现样本。
- `third` 用于存放第三方原始资料，不承载项目自己的研究内容或正式实现代码。

## 能力范围

### 新增 Capabilities

None.

### 修改的 Capabilities

- `project-governance`：补充项目目录结构及其职责划分要求。

## 影响范围

- 影响后续所有与策略开发、数据处理、回测、执行、测试和文档相关的变更。
- 影响后续对 `notebooks` 临时研究内容和 `third` 第三方资料的存放方式。
- 不直接引入运行时代码实现，但会作为后续实现的目录基线。
- 为未来创建具体策略模块和基础设施模块提供统一放置标准。
