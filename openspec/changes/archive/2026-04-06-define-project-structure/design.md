## 背景

量化项目在演进过程中通常会同时包含研究资料、数据管道、策略代码、回测引擎、执行适配和运行产物。如果这些内容缺少清晰的仓库边界，项目很快会出现“能跑但难维护”的问题，例如把实验笔记放进正式代码目录、把运行结果提交到实现目录、或者将策略实现和回测框架耦合在一起。

当前项目已经有较强的文档沉淀，因此目录设计应同时支持两类工作流：
- 面向研究的知识积累与策略说明。
- 面向工程的策略开发、回测验证和后续执行集成。

## 目标 / 非目标

**目标：**
- 为仓库定义一套稳定的顶层目录职责划分。
- 明确 `docs` 与 `src` 的分工，使研究内容与正式实现解耦。
- 为数据、配置、测试、脚本和输出产物建立标准落位。
- 为后续策略子模块提供可扩展的组织方式。
- 为数据资产和运行产物建立默认的 Git 提交边界。

**非目标：**
- 不在本次变更中实现回测引擎或策略代码。
- 不强制规定具体编程语言、框架或包管理方案。
- 不定义每个子目录下的全部文件命名细则，只定义职责边界和推荐结构。

## 关键决策

### Decision: 将目录结构标准纳入项目治理层
项目目录结构属于跨模块、长期生效的工程规则，因此将其定义在 `project-governance` capability 中，而不是只放在 README 或临时文档中。

Rationale:
- 目录边界会影响所有后续 change。
- 放入项目治理层后，策略、数据和执行相关变更都可以直接复用这套规则。
- 可以减少目录命名和职责划分上的反复讨论。

Alternatives considered:
- 单独新增 `project-structure` capability：可行，但当前规则更偏治理约束，直接纳入 `project-governance` 更简洁。

### Decision: 使用“研究区”和“实现区”双层结构
保留 `docs` 作为研究文档、方法说明和项目认知区，将 `src` 明确为正式实现区。

Rationale:
- 当前仓库已经在 `docs` 中积累了大量策略与市场相关内容。
- 将研究资产与可执行代码分离，有利于降低实验内容污染生产结构的风险。
- 更适合量化项目中“先研究、后实现、再验证”的迭代方式。

Alternatives considered:
- 将文档和代码全部收敛到单一目录：会增加职责混淆，不利于长期维护。

### Decision: `notebooks` 仅作为临时研究与验证区
将 `notebooks/` 视为临时研究、指标验证和想法试验的工作区，而不是正式实现或长期归档的主要承载位置。

Rationale:
- Notebook 适合快速探索和交互式验证，但通常不适合作为长期维护的生产实现形态。
- 明确其“临时性”可以减少核心逻辑长期滞留在 Notebook 中的情况。
- 研究沉淀应回流到 `docs/`，正式实现应沉淀到 `src/`。

Alternatives considered:
- 不单独定义 `notebooks/`：会让试验内容散落到 `docs/` 或 `src/`，增加边界混乱。

### Decision: `third` 专用于第三方原始资产
将 `third/` 明确为第三方官方文档、SDK、示例和供应商原始资料的承载区，不用于存放项目自己的研究内容、运行产物或正式实现代码。

Rationale:
- 第三方资料与项目自有资产应保持清晰边界。
- 便于识别“官方来源”和“项目内部结论”之间的差异。
- 有利于后续做依赖梳理、许可证检查和外部接口追踪。

Alternatives considered:
- 将第三方资料混放进 `docs/`：会削弱“内部认知”与“外部来源”的区分。

### Decision: 为策略开发保留完整工程闭环目录
顶层目录应至少覆盖数据、配置、正式代码、测试、脚本和输出，使项目从研究到回测再到执行具备完整落位。

推荐结构如下：

```text
ai-quant/
├─ docs/
├─ data/
│  ├─ raw/
│  ├─ interim/
│  ├─ processed/
│  └─ snapshots/
├─ notebooks/
├─ config/
│  ├─ env/
│  ├─ data/
│  └─ strategies/
├─ src/
│  ├─ common/
│  ├─ data/
│  ├─ factors/
│  ├─ strategies/
│  ├─ portfolio/
│  ├─ risk/
│  ├─ backtest/
│  ├─ execution/
│  └─ pipelines/
├─ tests/
│  ├─ unit/
│  ├─ integration/
│  └─ fixtures/
├─ scripts/
├─ outputs/
│  ├─ reports/
│  ├─ signals/
│  └─ logs/
└─ openspec/
```

Rationale:
- 这套结构覆盖量化项目的核心工程链路。
- 后续新增策略时可以在 `src/strategies` 下继续按风格或方法分类扩展。
- 回测和执行目录分离，便于保留“研究验证”和“交易落地”之间的安全边界。

Alternatives considered:
- 仅定义 `src/strategies` 和 `data` 两个核心目录：起步更轻，但很快会在测试、配置和输出管理上失去一致性。

### Decision: `data` 与 `outputs` 默认不纳入版本库
`data/` 和 `outputs/` 用于承载持续增长、可再生成或带有环境依赖的数据与运行产物，因此默认不提交 Git；仅允许提交目录占位文件以及为了测试或关键复现所保留的极小样本。

Rationale:
- 行情数据、中间结果和回测输出通常体积较大，不适合进入 Git 历史。
- 很多数据具有供应商授权、时效性或环境耦合，不宜作为常规版本资产管理。
- 保留小型测试样本在 `tests/fixtures/` 或经明确说明的小型快照中，更利于复现与 CI 使用。

Alternatives considered:
- 将全部数据纳入 Git：会快速增大仓库体积，也会降低协作体验。
- 完全禁止任何样本入库：会削弱测试与最小复现能力。

## 风险 / 权衡

- [Risk] 目录规划过早过细，可能与未来实际技术栈有偏差。 -> Mitigation: 规范只约束职责边界，不锁死实现细节；未来可通过新的 OpenSpec delta 微调。
- [Risk] 团队可能把推荐目录误解为“必须一次性全部创建”。 -> Mitigation: 在规范中强调这是标准落位，不要求同一时刻全部实现。
- [Risk] 旧文档与新规范之间可能出现表达不一致。 -> Mitigation: 后续与开发流程相关的项目文档应逐步对齐现行 spec。
- [Risk] 团队误把“少量样本可提交”扩展为“业务数据也可提交”。 -> Mitigation: 在规范中明确只允许测试夹具或关键复现实验的小规模样本，并默认忽略 `data/` 与 `outputs/`。
- [Risk] Notebook 中的试验逻辑长期未回流，导致实现与研究脱节。 -> Mitigation: 在规范中明确 `notebooks/` 为临时区，稳定结论回流到 `docs/` 或 `src/`。
- [Risk] `third/` 被当成“杂物目录”使用。 -> Mitigation: 在规范中限定其只存放第三方原始资产，并与 `docs/`、`src/`、`outputs/` 明确区分。

## 迁移计划

1. 在 `project-governance` 中加入项目目录结构要求。
2. 后续实现阶段按该标准逐步创建目录，而不是一次性填充空内容。
3. 新增策略、回测、执行和数据相关 change 时，默认遵循该目录职责边界。

## 开放问题

- 后续是否需要为 `notebooks` 制定更严格的清理或归档规则。
- 是否需要在下一次 change 中补充 Python 包布局、命名规范和配置加载约定。
