## Context

仓库当前已经提供 `src/data/a_share.py`，可以从 AkShare 获取、标准化并过滤 A 股股票列表；同时 Anne 主库已经具备 `data_sources`、`ingestion_batches`、`quality_checks` 与 `instruments` 基础表结构。缺口在于这两部分能力还没有被接起来，导致 A 股标的主数据无法形成可重复执行、可审计的正式入库流程。

本次设计需要在不修改基础 schema 的前提下，新增一条围绕 A 股股票列表的入库通道。该通道要把抓取结果映射到 `instruments` 表，并把来源、批次和质量检查纳入统一治理结构，便于后续市场数据、财报数据和事件数据直接复用已落库的证券主数据。

约束包括：

- 复用现有 AkShare 抓取与标准化逻辑，不重新定义股票列表口径。
- 与现有 `instruments(exchange, symbol)` 唯一约束兼容，避免重复插入。
- 入库流程需要支持重复执行，且重复执行时结果可预期。
- 第一版聚焦 A 股股票列表入库，不扩展到退市补齐、上市状态回补或多资产类型统一导入。

## Goals / Non-Goals

**Goals:**

- 建立从 A 股股票列表到 `instruments` 的标准入库流程。
- 在入库时自动维护 `data_sources` 与 `ingestion_batches` 记录，并写入批次级质量检查结果。
- 定义字段映射、幂等 upsert 规则和默认治理字段填充方式。
- 提供脚本入口，支持手动执行入库并输出摘要结果。
- 为入库行为提供测试覆盖，确保映射、去重和幂等逻辑稳定。

**Non-Goals:**

- 不在本次设计中新增或修改 Anne 核心表 schema。
- 不引入新的调度系统、任务队列或 Airflow 工作流。
- 不扩展到指数、基金、港股、美股或其他资产列表入库。
- 不在第一版中处理复杂的退市状态、停牌状态或行业分类回填。

## Decisions

### 1. 采用“抓取模块 + 入库模块 + 脚本入口”三层结构

选择在现有 `src/data/a_share.py` 之上新增专门的入库模块，由脚本入口调用入库服务，而不是把数据库写入逻辑直接混入抓取模块或脚本中。

原因：

- 抓取与标准化逻辑已经独立存在，继续保持单一职责更利于测试和复用。
- 入库模块可以单独处理数据库连接、事务、upsert 和治理表写入。
- 脚本只负责参数接收和结果展示，后续也更容易接入定时任务。

备选方案：

- 直接在现有脚本中拼接 SQL：实现快，但可维护性差，测试边界不清晰。
- 把数据库逻辑写进 `a_share.py`：会把“数据获取”和“持久化”耦合在一起。

### 2. `instruments` 写入采用按 `(exchange, symbol)` 幂等 upsert

选择以 `exchange + symbol` 作为业务主身份，对应现有唯一约束，在重复执行入库时更新名称、类型、状态、元数据和治理字段，而不是盲目插入新记录。

原因：

- 该唯一键已经在 schema 中明确存在，符合当前数据库设计。
- A 股股票列表是快照型主数据，重复执行更适合表现为“刷新当前标准记录”。
- 幂等 upsert 可降低日常刷新任务的失败风险。

备选方案：

- 每次入库生成新版本记录：会快速堆积重复主数据，且与现有唯一约束冲突。
- 先查后插再更新：逻辑更繁琐，批量性能和事务一致性也更差。

### 3. 治理记录按“单次脚本执行一批次”建模

选择每次执行股票列表入库时创建或复用一个 `data_sources` 记录，并生成一条新的 `ingestion_batches` 记录，所有本次写入的 `instruments` 记录统一挂到该批次上。

原因：

- 股票列表是批量快照数据，天然适合按批次治理。
- 后续质量检查、失败排查和数据追溯都需要批次视角。
- 与现有治理表设计天然一致，不需要新增额外建模。

备选方案：

- 不记录批次，只更新 `instruments`：追溯能力不足，不符合 Anne 治理目标。
- 每条 `instrument` 单独建批次：粒度过细，没有实际收益。

### 4. 第一版质量检查聚焦记录数、重复键和空关键字段

选择在入库流程中至少记录三类批次级检查：标准化结果非空、`exchange + symbol` 无重复、`symbol/name/exchange` 关键字段无空值。检查结果写入 `quality_checks`。

原因：

- 这些检查能覆盖主数据入库最常见的结构性错误。
- 检查规则简单、稳定，适合作为第一版基线。
- 结果写入治理表后，可为后续更细粒度的数据质量规则留出扩展点。

备选方案：

- 不做质量检查：入库成功但数据质量不可见。
- 一开始引入复杂业务校验：实现成本偏高，且超出本次范围。

## Risks / Trade-offs

- [AkShare 上游字段或口径变化导致映射失效] -> 继续复用现有标准化模块并以测试锁定映射预期，避免在入库层重复猜测字段。
- [批量 upsert 更新范围过大，覆盖人工修正字段] -> 第一版仅更新股票列表直接提供或明确受控的字段，避免覆盖未来可能的人工补录字段。
- [质量检查过于基础，无法识别业务异常标的] -> 先记录结构性检查结果，后续再按真实问题扩展规则。
- [重复执行会刷新 `batch_id` 与治理时间] -> 接受这是“最新一次标准化快照”的语义，历史批次通过治理表保留。

## Migration Plan

1. 在代码层新增 A 股股票列表入库模块，封装数据源登记、批次创建、质量检查和 `instruments` upsert。
2. 新增脚本入口，支持手动执行 A 股股票列表入库，并输出插入数、更新数和检查结果摘要。
3. 为入库模块和脚本补充测试，验证幂等行为与治理记录逻辑。
4. 在本地 Anne PostgreSQL 环境执行一次端到端验证，确认相关表数据符合预期。

回滚策略：

- 代码回滚可直接回退本次变更。
- 数据回滚以批次为边界，必要时可根据新增批次记录定位并清理本次写入的 `instruments` 记录或恢复字段值。

## Open Questions

- `instruments.status` 第一版是否统一写为 `active`，还是需要为未知状态保留更中性的默认值。
- `data_sources.source_code` 是否固定为类似 `akshare_a_share_stock_list` 的稳定编码，供后续其他 AkShare 入库任务复用。
