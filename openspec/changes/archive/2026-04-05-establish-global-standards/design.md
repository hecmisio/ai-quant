## 背景

当前仓库已经使用 OpenSpec，但尚未定义项目级全局规范，无法统一约束后续能力应如何提案、组织、实现和审查。由于本项目定位为 AI 辅助的量化研究与交易系统，因此架构边界和风险控制的重要性，明显高于一般信息系统。

本设计通过两个互补的 capability 建立一层长期有效的治理基线：
- `project-governance`：负责仓库级工程规范与交付规则
- `ai-team-operating-model`：负责 AI 协作团队的角色分离与交接规则

设计目标是在保证约束清晰的同时，避免把治理做成沉重的流程负担。

## 目标 / 非目标

**目标：**
- 为项目建立稳定的架构、流程与安全边界基线。
- 将四角色协作模式沉淀为正式规范，而不是仅停留在说明性文档中。
- 强制要求市场数据、信号生成、风险控制、交易执行之间保持清晰分层。
- 为后续 OpenSpec 变更提供统一的审查基线，降低跨模块变更时的理解成本。

**非目标：**
- 不实现任何运行时交易模块、券商适配器或执行引擎。
- 不提前绑定具体编程语言、框架或存储后端。
- 不定义具体的量化策略细节或参数值。
- 不替代未来大型功能变更所需的独立 design 工作。

## 关键决策

### Decision: 使用长期 spec 承载全局规范
项目级规则将以标准 OpenSpec capability 的形式落在 `openspec/specs/` 中，而不是只写在说明文档里。

Rationale:
- 全局规范应当独立于单次 change，长期生效。
- 后续变更可以通过标准 OpenSpec 流程引用或修改这些能力。
- 这样可以让治理规则具备可见性、可审查性和可归档性。

Alternatives considered:
- 仅写在 README 或 docs 中：初期更简单，但缺乏变更跟踪和约束力。
- 仅写在代码注释中：过于分散，不适合作为项目级规则来源。

### Decision: 将治理规范与协作模式分成两个 capability
本次变更将项目级规范拆分为两个 capability，而不是合并成一个。

Rationale:
- 项目治理规则和团队协作规则的演化速度不同。
- 即使未来角色命名调整，架构和流程规则仍然可以保持稳定。
- 拆分后，后续增量修改会更清晰，影响面也更容易控制。

Alternatives considered:
- 合并为单一 capability：创建更快，但会混合不同层次的约束，后续维护会更混乱。

### Decision: 将风控闸门设为强制边界
规范要求所有可执行订单都必须经过显式风控审批，不能从策略层直接进入执行层。

Rationale:
- 风控闸门是交易系统中的基础安全不变量。
- 它能强制保留从信号到执行之间的审批边界，提升可审查性。
- 这也与四角色协作模式保持一致。

Alternatives considered:
- 允许策略模块在实验阶段直接调用执行接口：虽然开发更快，但风险极高，且容易把实验路径带入正式系统。

### Decision: 规范保持强约束，但不绑定具体技术实现
spec 使用 MUST 级别要求来定义行为和流程，但避免在现阶段绑定过细的技术选型。

Rationale:
- 当前项目仍处于实验阶段，后续可能会调整语言、框架或底层组件。
- 现阶段最重要的是边界、职责和流程一致，而不是提前锁死实现方式。

Alternatives considered:
- 现在就规定完整技术栈：灵活性太低，不适合当前探索阶段。

## 风险 / 权衡

- [Risk] 规范过于抽象，未来执行时可能缺少操作性。 → Mitigation: 使用可测试的 requirement 和 scenario 表达，并在后续 delta 中持续细化。
- [Risk] 全局规范可能让小实验显得流程过重。 → Mitigation: 仅约束边界、流程和安全底线，不下沉到实现细节。
- [Risk] 团队角色模型可能被误解为强制代码结构。 → Mitigation: 明确其含义是职责与交接规则，而不是类名或模块名。
- [Risk] 后续贡献者可能忽略这些规则。 → Mitigation: 将其落入 OpenSpec，后续 change 审查时可直接对照 capability 检查。

## 迁移计划

1. 通过本次 change 引入两个新的项目级 capability。
2. 将该 change 归档，使两个 capability 正式进入 `openspec/specs/`。
3. 未来所有涉及数据、策略、风控、执行和项目文档的 change，都以这套规范为基线。
4. 当仓库规模继续扩大时，再通过后续 delta 增补更细的目录规范、测试门槛或发布规则。

## 开放问题

- 后续是否需要增加第三个 capability，专门约束研究证据、回测结果和评估材料。
- 团队角色名称在正式规范中是否始终保留拟人化命名，还是逐步转为纯职责命名。
- 执行相关功能未来是否必须先经过仿真阶段，再允许进入真实券商接口。
