## Context

当前仓库已经通过 `BaseStrategy`、`MACDStrategy`、研究脚本与回测模块打通了一个最小策略闭环，但这个闭环目前只覆盖趋势跟随场景。网格策略与 MACD 在信号机制上明显不同：它更依赖价格区间、离散层级和分段仓位，而不是连续指标交叉，因此很适合验证现有策略接口是否足够通用。

仓库现状也带来几个约束：

- `BaseStrategy` 只提供 `prepare_data`、`generate_signals`、`build_positions` 和 `get_params` 这组最小接口。
- 现有策略实现以 `pandas.DataFrame` 为输入输出载体，并默认保留原始索引和行顺序。
- 当前回测层偏向读取策略产出的 `signal` 与 `target_position`，不适合在首版网格策略里引入复杂的订单簿、挂单状态或成交回报模拟。

因此，这次设计需要把网格策略定义为一个“可直接从价格映射出目标仓位”的离散仓位策略，而不是依赖外部执行引擎的交易系统。

## Goals / Non-Goals

**Goals:**

- 定义一个与 `BaseStrategy` 兼容的首版 `GridStrategy`，接受标准行情表并输出网格信号与目标仓位。
- 明确网格策略的核心参数：`lower_bound`、`upper_bound`、`grid_count`、`price_column`。
- 使用固定价格区间和等距网格生成离散仓位层级，使策略行为可预测、可测试、可回测接入。
- 让价格跨越多个网格时也能在单根 bar 上得到确定性的目标仓位结果。

**Non-Goals:**

- 不实现动态再中心化、自动扩展区间或跟随趋势移动网格中心。
- 不实现做空网格、对冲网格或马丁加仓等高风险扩展。
- 不在首版引入真实挂单簿、部分成交、滑点路径依赖或逐笔撮合状态机。
- 不修改现有回测协议为网格策略定制专门执行层。

## Decisions

### 1. 继续沿用 `BaseStrategy + DataFrame` 契约

网格策略将保持与 `MACDStrategy` 相同的使用方式：输入为按时间顺序排列的 `pandas.DataFrame`，默认依赖 `close` 列，输出继续在原表基础上附加计算字段。

Rationale:

- 这能让首版网格策略复用现有测试、脚本和回测接入方式。
- 当前仓库没有更复杂的市场数据抽象，继续使用 DataFrame 成本最低。

Alternatives considered:

- 引入单独的策略输入对象或行情 DTO：抽象更强，但会放大改动面，不适合当前阶段。

### 2. 使用固定区间 + 等距网格生成价格层级

策略参数中显式要求 `lower_bound < upper_bound`，并使用 `grid_count` 表示区间被切分的网格份数。系统据此生成 `grid_count + 1` 个价格层级，网格间距为 `(upper_bound - lower_bound) / grid_count`。

Rationale:

- 显式区间和等距网格更容易理解，也最容易写成稳定测试。
- 固定层级可以直接映射离散仓位比例，避免将实现绑定到“先挂单再成交”的执行模型。

Alternatives considered:

- 按百分比或等比网格生成层级：更灵活，但初版参数理解和测试复杂度更高。
- 由基准价自动推导上下界：对用户更省事，但会引入额外隐式假设。

### 3. 目标仓位由“当前价格所在网格区间”直接决定

系统不会维护一个独立的挂单状态机，而是根据当前价格落在哪个网格区间，直接计算 `target_position`。当价格越低，目标仓位越高；当价格越高，目标仓位越低。区间外采用饱和处理：价格低于下界时目标仓位为 `1.0`，价格高于上界时目标仓位为 `0.0`。

Rationale:

- 直接映射仓位让策略在任何单行输入上都可重复计算，行为完全确定。
- 跳空一次跨越多格时无需循环补单，直接得到最终仓位，更适合研究和回测场景。

Alternatives considered:

- 维护“每穿越一格成交一笔”的状态机：更贴近实盘挂单，但会把首版实现推向执行引擎问题。

### 4. 信号由目标仓位变化方向派生

`generate_signals` 将在结果中暴露网格层级、当前网格索引和建议信号。若当前 `target_position` 高于上一行则为 `buy`，低于上一行则为 `sell`，相等则为 `hold`。这让信号语义与最终仓位保持一致。

Rationale:

- 对网格策略来说，“加仓/减仓”比单次买卖更本质，信号跟随目标仓位变化最自然。
- 这样可以与现有回测层消费的 `signal`、`target_position` 字段保持兼容。

Alternatives considered:

- 单独设计一套 `rebalance_up` / `rebalance_down` 信号：语义更细，但会增加对下游代码的适配成本。

### 5. 首版只做长仓网格与输入校验

实现只允许 `target_position` 处于 `[0.0, 1.0]` 区间，并使用固定步长 `1 / grid_count` 变化。输入缺少价格列、上下界非法、`grid_count < 1` 或价格不可解析时，系统必须明确报错或在单行上安全退化。

Rationale:

- 长仓网格已经足够覆盖“跌了买、涨了卖”的基础策略认知。
- 清晰参数校验和异常处理比追求功能覆盖更重要。

Alternatives considered:

- 同时支持双向持仓和自定义每格仓位：扩展性更强，但会显著增加规格和测试面。

## Risks / Trade-offs

- [Risk] 固定上下界在单边行情中容易长期满仓或空仓。 -> Mitigation: 在规格中明确边界饱和行为，并把动态再中心化留到后续独立变更。
- [Risk] 直接映射目标仓位弱化了真实挂单成交过程，回测结果会比实盘更理想化。 -> Mitigation: 将该策略定位为研究型信号层，后续若需要执行 realism，再单独扩展执行模块。
- [Risk] 网格参数对结果高度敏感，不同标的和周期下容易产生误导性结论。 -> Mitigation: 在实现中暴露完整参数并补足测试，让研究者能明确知道策略假设。
- [Risk] 现有回测模块目前偏向 MACD 使用路径，网格策略接入时可能需要少量通用化调整。 -> Mitigation: 保持输出字段命名与现有策略一致，优先复用而非重写回测约定。

## Open Questions

- 是否在首版就增加独立的 `scripts/run_grid_strategy.py`，还是先以 Python API 和测试用例验证策略能力。
- 后续如果要做实盘风格网格，应该以“执行层扩展”还是“策略状态机扩展”方式演进。
