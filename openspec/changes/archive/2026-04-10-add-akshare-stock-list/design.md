## Context

后续无论是股票池构建、批量回测，还是为 Anne 准备 A 股标的主数据，都需要一个统一的 A 股股票列表来源。当前仓库尚未提供这类基础能力，因此每次需要股票全集时都会缺少稳定、可复用的实现入口。

本次设计目标是基于 AkShare 获取 A 股股票列表，并在第一版中默认过滤掉 ST 和 *ST 股票。这个能力应服务于数据层和脚本层，而不是只停留在临时 notebook 中。

约束包括：

- 使用 AkShare 作为数据来源。
- 输出必须标准化为仓库内部可复用的字段命名。
- 默认过滤规则聚焦 ST / *ST，不扩展到停牌、退市整理、上市天数等更复杂规则。
- 实现需要既能被 Python 代码复用，也能通过命令行脚本触发。

## Goals / Non-Goals

**Goals:**

- 提供一个稳定的 A 股股票列表获取函数。
- 对返回字段做标准化，至少包含股票代码、名称和市场信息。
- 默认排除 ST 与 *ST 股票。
- 提供脚本入口，便于手工执行和后续流水线接入。
- 为测试提供可验证的过滤规则和最小行为约束。

**Non-Goals:**

- 不在本次实现中定义更复杂的股票池规则，如上市天数、流动性、行业、停牌状态。
- 不将股票列表直接写入 PostgreSQL 或 Anne 主库。
- 不保证 AkShare 上游字段永远不变，本次只在当前接口基础上做标准化封装。
- 不覆盖港股、美股或基金列表。

## Decisions

### 1. 采用“数据函数 + 脚本入口”双层结构

选择在 `src/data/` 提供可导入的获取函数，在 `scripts/` 提供一个命令行入口。

原因：

- 策略、数据管道和测试更适合复用函数，而不是直接依赖脚本。
- 脚本入口方便手工运行、导出和后续任务编排。

备选方案：

- 只提供脚本：复用性差。
- 只提供函数：不利于快速验证和命令行操作。

### 2. ST 过滤以股票名称标签为准

第一版选择基于股票简称中的 `ST` 与 `*ST` 标记进行默认过滤。

原因：

- 对 AkShare 返回结果来说，这是成本最低且最直接的第一版规则。
- 满足“默认排除 ST 股”的用户需求，不需要引入额外数据源。

备选方案：

- 基于更复杂的状态字段或公告规则过滤：准确性可能更高，但超出本次范围。

### 3. 输出字段先收敛到最小稳定集合

第一版标准输出至少包含：

- `symbol`
- `name`
- `exchange`
- `market`

如 AkShare 提供其他字段，可保留但不把它们作为首版契约核心。

原因：

- 先保证通用性和稳定性。
- 避免上游字段轻微变化导致首版能力过度脆弱。

## Risks / Trade-offs

- [AkShare 上游接口字段变化] -> 在实现中集中做字段映射，并用测试锁定预期输出列。
- [仅按名称过滤 ST 可能漏掉边缘状态] -> 在设计中明确这是第一版规则，后续可增量扩展。
- [部分股票简称包含 ST 字样但并非监管意义的 ST] -> 先用常见规则满足基础需求，后续根据真实问题再细化。

## Migration Plan

1. 在 `src/data/` 新增 A 股股票列表获取与过滤模块。
2. 在 `scripts/` 新增命令行脚本，用于输出过滤后的股票列表。
3. 增加测试，验证字段标准化和 ST 过滤行为。
4. 后续如需写入数据库或接入更复杂股票池规则，再通过新 change 扩展。
