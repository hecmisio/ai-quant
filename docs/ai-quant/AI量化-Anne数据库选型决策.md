# AI量化-Anne数据库选型决策

## 一、背景

在 AI 量化团队协作模型中，情报官 Anne Hathaway 的职责不是单纯保存行情数据，而是负责市场数据的采集、清洗、标准化与交付，确保下游分析、风控和执行使用的是可信、可追溯、可复用的数据。

这意味着 Anne 面对的数据天然是混合型的，而不是单一结构：

- 行情数据：K 线、分时、逐笔、盘口快照
- 财报数据：利润表、资产负债表、现金流量表、业绩预告、业绩快报
- 宏观经济数据：CPI、PPI、PMI、社融、利率、汇率、就业、库存等时间序列
- 新闻事件与分析：新闻正文、摘要、标签、情绪、事件类型、来源可信度
- 机构研报：报告元信息、评级变动、目标价、行业观点、PDF 及文本切片
- 财经日历：财报披露日、分红送转、宏观数据发布时间、央行会议、解禁日等
- 大模型驱动催化剂事件：LLM 从新闻、公告、研报中抽取的事件、实体关系、影响方向、置信度和推理摘要

因此，本次数据库选型的核心问题不是“哪种数据库最快”，而是“哪种存储体系最适合 Anne 作为统一数据就绪层的职责定位”。

## 二、决策目标

数据库选型需要同时满足以下目标：

- 支持时间序列数据的稳定写入与查询
- 支持半结构化和异构数据的统一建模
- 支持事件、文档、实体、标的之间的关联分析
- 支持后续接入大模型的语义检索、事件抽取和催化剂识别
- 支持 Python / pandas 生态，降低研究与工程集成成本
- 支持数据质量治理、版本追踪和可追溯性
- 允许项目从个人研究阶段平滑扩展到团队化和服务化阶段

## 三、候选方案

本次主要比较以下几类方案：

### 方案 A：PostgreSQL 作为主库

特点：

- 关系模型成熟，事务能力强
- 原生支持复杂查询、关联分析和数据约束
- `JSONB` 可容纳半结构化字段
- Python 生态成熟，易于与 `pandas`、`SQLAlchemy`、`polars` 协同
- 可进一步扩展 `TimescaleDB`、`pgvector`

适合：

- 需要统一管理结构化、半结构化、事件型数据
- 需要为后续研究、风控、审计提供一致数据源
- 希望在一个主库中先解决 80% 问题

### 方案 B：ClickHouse 作为主库

特点：

- 列式存储，聚合分析性能极强
- 适合大规模行情、日志、埋点和宽表分析
- 对高吞吐时序分析和统计报表很有优势

适合：

- 海量 tick、盘口、因子、日志数据
- 以分析吞吐和压缩比为第一优先级

局限：

- 事务和约束能力弱于 Postgres
- 文档、事件、治理类建模不如 Postgres 自然
- 作为早期统一事实源，维护复杂度偏高

### 方案 C：MongoDB 作为主库

特点：

- 文档模型灵活，适合异构数据快速落地
- 开发初期对字段变化容忍度高

适合：

- 以 JSON 文档为主、结构频繁变化的场景

局限：

- 时间序列分析、复杂关联、严格数据治理能力不占优
- 对量化研究常见的联表、窗口分析和口径统一不够理想

### 方案 D：DuckDB / Parquet 作为主存储

特点：

- 本地分析体验很好
- 对列式文件读取和研究型分析非常高效
- 与 Python 集成成本低

适合：

- 单机研究、离线回测、分析缓存

局限：

- 不适合作为多源采集后的统一在线事实库
- 不擅长做多角色协作下的数据治理和服务化访问

### 方案 E：多库组合

典型组合：

- Postgres 负责事实库与治理
- 对象存储或文件系统负责原始文件
- DuckDB / Parquet 负责离线分析
- 后期如数据量暴增，再引入 ClickHouse 做分析加速

这种方案更接近真实的量化数据基础设施。

## 四、关键判断

### 1. Anne 需要的是“统一事实源”，不是单一行情库

如果只处理日线 K 线，很多数据库都能胜任。但 Anne 还要承接新闻、研报、日历、财报和 LLM 事件抽取结果，这决定了主库必须同时擅长：

- 时间序列
- 事件建模
- 文档元数据
- 实体关系
- 半结构化字段
- 可审计的数据治理

在这一点上，PostgreSQL 的平衡性最好。

### 2. 原始文件不应全部硬塞进主库

研报 PDF、新闻原文 HTML、公告附件、原始 JSON 响应、批量 CSV 文件更适合放在文件系统或对象存储中，再把元信息、解析结果和索引写入数据库。

这样做有几个好处：

- 主库更聚焦于可查询事实，而不是承担大文件仓库职责
- 原始文件保留更完整，便于重新解析和回溯
- 后续迁移存储介质时成本更低

### 3. 大模型催化剂事件不是独立数据库问题，而是数据模型问题

很多团队一想到 LLM，就倾向于先上向量数据库。这个顺序通常不对。

对 Anne 来说，更重要的是先定义好：

- 事件是什么
- 事件关联哪些标的、行业、主题、国家、宏观变量
- 事件有何方向、强度、时效性、置信度
- 事件由哪些原始文档或新闻支持

这些数据首先应有稳定的关系型事实模型。向量检索只是检索增强层，而不是主数据层。

## 五、最终决策

### 结论

Anne 的主数据库优先选择：

- `PostgreSQL`

建议从一开始就为后续能力预留扩展：

- 时间序列扩展：`TimescaleDB`
- 半结构化字段：`JSONB`
- 向量检索：`pgvector`

### 决策理由

选择 PostgreSQL 的主要原因有：

- 对混合型金融数据的适配最均衡
- 能同时处理关系型表、事件表和半结构化字段
- 易于表达数据约束、唯一性、血缘和审计信息
- 与 Python 研究栈兼容性高，适合当前仓库以 `pandas.DataFrame` 为中心的工作方式
- 后续扩展到向量检索和时序分区时路径清晰
- 作为统一事实源，比 ClickHouse 或 MongoDB 更适合项目早期阶段

## 六、不选择其他方案作为第一主库的原因

### 不把 ClickHouse 作为第一主库

不是因为 ClickHouse 不强，而是因为它更适合在以下阶段发力：

- tick / quote / orderbook 数据量爆炸
- 需要超大规模因子统计和分钟级聚合分析
- 需要高吞吐分析查询，而不是复杂治理与建模

在 Anne 的第一阶段建设中，核心矛盾不是“分析速度不够”，而是“数据种类多、口径复杂、需要统一治理”。这不是 ClickHouse 的主战场。

### 不把 MongoDB 作为第一主库

MongoDB 在新闻、研报和事件文档落地上很方便，但它会诱导系统走向“先堆文档，后补治理”的路径。对于量化系统，这通常会导致：

- 字段口径逐渐漂移
- 跨数据源关联成本上升
- 下游研究表和风控口径难统一

Anne 的职责要求可交付、可复用、可追溯，这更适合以关系型事实模型为主。

### 不把 DuckDB / Parquet 作为唯一存储

DuckDB 和 Parquet 很适合作为研究和分析层，但不适合作为唯一主库。原因在于：

- 它们更像分析格式，不像协作系统的权威事实源
- 写入治理、并发访问、任务编排和审计支撑较弱
- 多源摄入后不易形成稳定的数据服务边界

## 七、建议的存储分层

建议采用“四层式”方案：

### 第 1 层：Raw Layer

存放原始文件和原始响应：

- CSV
- JSON
- PDF
- HTML
- 图片
- 抓取快照

可先使用：

- 仓库内 `data/raw`

后续升级为：

- MinIO
- S3 兼容对象存储

### 第 2 层：System of Record

使用 `PostgreSQL + TimescaleDB` 存放标准化后的权威数据：

- 行情事实表
- 财报事实表
- 宏观序列表
- 新闻与研报元数据表
- 事件事实表
- 日历事件表
- 数据源表
- 入库批次表
- 质量检查表

### 第 3 层：Search / RAG Layer

使用 Postgres 内的 `pgvector` 支持：

- 新闻相似检索
- 研报切片语义召回
- 历史催化剂事件相似匹配

这一步不必独立部署外部向量库，除非后期规模和性能要求明显上升。

### 第 4 层：Research / Cache Layer

使用 `Parquet + DuckDB` 或本地缓存文件支持：

- 回测数据集抽取
- 因子实验
- 快速宽表分析
- notebook 临时研究

这一层服务 Sophie 和研究型工作负载，不承担主事实源职责。

## 八、推荐的阶段性落地路线

### 第一阶段：现在就该做的

- 确定 `PostgreSQL` 为 Anne 主库
- 原始文件继续保存在 `data/raw` 或独立原始存储目录
- 为行情、财报、宏观、新闻、研报、日历、催化剂事件设计统一主键和来源字段
- 所有表都保留 `source`、`ingested_at`、`as_of`、`version`、`raw_uri` 等治理字段

### 第二阶段：数据量上来后

- 为高频时序表引入 `TimescaleDB hypertable`
- 为新闻、研报、事件切片引入 `pgvector`
- 增加数据质量校验和重放机制

### 第三阶段：规模显著扩大后

- 如果 tick / orderbook / 因子宽表规模暴增，再引入 `ClickHouse`
- 将 ClickHouse 定位为分析加速层，而不是替代 Postgres 的主事实源

## 九、最终建议

如果目标是为 Anne 打一个可持续扩展的数据底座，最合适的选型不是“只选一个数据库解决所有问题”，而是：

- 主库选 `PostgreSQL`
- 时序增强选 `TimescaleDB`
- 语义检索增强选 `pgvector`
- 原始文件放文件系统或对象存储
- 研究分析层使用 `Parquet + DuckDB`

用一句话总结这次决策：

Anne 的第一性需求是“统一、可治理、可扩展的数据事实源”，因此主库应选择 PostgreSQL，而不是以单一时序性能或单一文档灵活性为中心的数据库。
